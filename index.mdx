Lynx helps you see what happened during an AI agent run. Connect the SDK to record sessions, trace each execution step, inspect LLM and tool calls, and review root cause candidates when something fails.

Use Lynx when logs are not enough to answer questions like:

- What did the user ask?
- Which context did the agent load?
- Which LLM and tool calls happened?
- Where did the run fail or slow down?
- Did a guardrail block a risky tool call?

## What Lynx records

The SDK can record:

- Session, run, workspace, agent, and environment metadata
- User input and selected context
- Agent decisions and supporting reasons
- LLM calls, model metadata, latency, token usage, and errors
- Tool calls, tool results, side effect metadata, and risk level
- Local policy decisions and guardrail actions
- Memory access and final run outcome

## What you can see in Lynx

After events are sent, Lynx can show:

- The full session timeline
- Each LLM call and tool call
- Latency, token usage, and estimated cost
- Errors, retries, and slow execution steps
- Guardrail decisions and blocked tool calls
- Root cause candidates with supporting evidence

## Production behavior

By default, SDK delivery is background-only. If the Lynx API is unavailable, your service keeps running. The SDK queues events in memory, retries failed sends, and opens a circuit breaker after repeated failures.

Guardrail checks run locally in your process. Lynx API availability does not decide whether a guarded tool call is blocked.

<Note>
  Lynx may receive prompts, tool arguments, and model responses depending on
  your SDK configuration. Avoid sending secrets or credentials. Use masking or
  `metadata-only` mode when you do not want raw content stored.
</Note>
