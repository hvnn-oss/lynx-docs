Lynx SDK는 수집 서버 장애가 고객 서비스의 요청 실패로 이어지지 않도록 설계되어 있습니다. 기본값에서는 이벤트를 메모리에 큐잉하고, 짧은 timeout과 circuit breaker로 전송 실패를 격리합니다.

## 운영 기본값

| 옵션                              | 기본값          |
| --------------------------------- | --------------- |
| `delivery.mode`                   | `"BACKGROUND"`  |
| `delivery.timeoutMs`              | `1000`          |
| `delivery.flushOnRunEnd`          | `false`         |
| `delivery.flushIntervalMs`        | `3000`          |
| `delivery.batchSize`              | `50`            |
| `delivery.maxQueueSize`           | `1000`          |
| `delivery.overflowStrategy`       | `"DROP_OLDEST"` |
| `circuitBreaker.enabled`          | `true`          |
| `circuitBreaker.failureThreshold` | `3`             |
| `circuitBreaker.cooldownMs`       | `30000`         |

## 백그라운드 전송

백그라운드 모드에서 `run()`은 이벤트를 로컬 큐에 기록하고 전송을 기다리지 않습니다. SDK는 주기적으로 큐를 flush하며, 명시적으로 `flush()`나 `shutdown()`을 호출할 수도 있습니다.

```ts
const lynx = new LynxTracer({
  clientId: "support-api",
  apiKey: process.env.LYNX_API_KEY,
  delivery: {
    mode: "BACKGROUND",
    timeoutMs: 1000,
    flushOnRunEnd: false,
  },
});
```

## Blocking 전송

`BLOCKING` 모드는 batch size에 도달했을 때 즉시 flush를 시도할 수 있습니다. 그래도 네트워크 실패는 애플리케이션 예외로 전파하지 않고 SDK 내부에서 처리합니다.

```ts
const lynx = new LynxTracer({
  clientId: "support-api",
  apiKey: process.env.LYNX_API_KEY,
  delivery: {
    mode: "BLOCKING",
    batchSize: 20,
    timeoutMs: 1000,
  },
});
```

## flush와 shutdown

`flush()`는 현재 큐에 쌓인 이벤트를 전송합니다.

```ts
await lynx.flush();
```

`shutdown()`은 서버 종료, CLI 종료, serverless handler 종료처럼 런타임이 곧 내려갈 때 사용합니다.

```ts
await lynx.shutdown({ timeoutMs: 1000 });
```

## Lynx 서버가 내려가면

Lynx 서버가 내려가도 고객 서비스는 계속 실행됩니다.

- 전송 실패는 SDK 내부에서 잡습니다.
- 실패한 이벤트는 retry queue에 보관합니다.
- 연속 실패가 많아지면 circuit breaker가 열립니다.
- 큐가 꽉 차면 `overflowStrategy`에 따라 오래된 이벤트나 새 이벤트를 버립니다.

가드레일 판단은 SDK의 로컬 정책 평가를 우선합니다. 서버가 내려가도 이미 캐시되어 있거나 코드로 정의된 정책은 계속 평가됩니다.
